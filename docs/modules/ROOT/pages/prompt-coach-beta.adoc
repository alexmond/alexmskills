= Prompt Coach (beta)
:navtitle: prompt-coach-beta

A `UserPromptSubmit` hook that watches every prompt sent to Claude Code and nudges toward better prompting habits ‚Äî definition-of-done, scoped references, guardrails, verification, and more. Rules quietly graduate once you consistently apply them, so the coach fades as your prompts improve.

The `-beta` suffix is intentional: the plugin ships from the same marketplace as the stable plugins but is signposted as in-progress. It graduates to `prompt-coach` when it earns its stable slot.

== Install

[source,text]
----
/plugin marketplace add alexmond/alexmskills
/plugin install prompt-coach-beta@alexmskills
/reload-plugins
----

That's it. No separate channel, no extra configuration. Restart the session so the `UserPromptSubmit` hook registers.

== What it watches for

Six tiers of rules, roughly five each. Only six rules are active at a time (`max_active_rules`, default 6) ‚Äî lower tiers dominate, and as a rule masters, the next dormant rule activates in its place. When all rules in a tier master, the next tier's rules take their place.

=== L1 ‚Äî fundamentals

[cols="1,3",options="header"]
|===
| Rule | Catches
| vague-reference | "Fix it" / "improve this" with no named file/function
| no-definition-of-done | Action verb with no acceptance criteria
| unbounded-scope | "all / every / entire X" attached to a mutating verb
| improve-without-metric | "better / faster / cleaner" with no measurable target
| missing-guardrails | Heavy verb (refactor/rewrite/migrate) with no "don't touch X"
| no-answer-shape | "What are X" / "how much of Y" question without a format spec
|===

=== L2 ‚Äî intermediate

[cols="1,3",options="header"]
|===
| Rule | Catches
| compound-tasks | Three+ action verbs joined by "and"
| no-verify-loop | Implementation ask with no verification step
| missing-context-fetch | "The failing test / the issue" with no identifier
| no-format-spec | Ask for summary/list/report with no shape
|===

=== L3 ‚Äî classical prompting techniques

[cols="1,3",options="header"]
|===
| Rule | Catches
| no-adversarial-check | High-stakes ask (security/migration/prod/delete) with no skeptic
| retry-without-diagnosis | Short "try again" with no new information
| no-few-shot | "Like X" / "in the style of Y" without an example
| no-chain-of-thought | Reasoning ask (why/debug/trace) without "think first"
| no-rubric | Judgment ask ("is this good?") without criteria/axes
| no-uncertainty-budget | Investigative ask with no "if unsure, say so"
| no-xml-structure *(v0.20+)* | ‚â•7-line paste (code fence or indented block) without `<code>` / `<data>` / `<logs>` tags
| no-classical-role *(v0.20+)* | Review / audit / critique ask (>15 words) without "You are a X" / "act as"
| test-goalseeking *(v0.20+)* | "Make tests pass" / "get CI green" without a stated correctness intent
| no-verify-before-claim *(v0.20+)* | "Does X exist?" / "which files use Y?" without demanding file:line receipts
|===

=== L4 ‚Äî goals & loops

[cols="1,3",options="header"]
|===
| Rule | Catches
| implicit-goal | Action prompt with no "so that / in order to / why" clause
| unbounded-iteration | "Keep improving" without a stopping condition
| no-rubric-for-refine | "Refine this" without naming which axis to improve
| overthinking-warning *(v0.20+)* | ‚â•60-word prompt with 3+ "make sure to / be very careful / please also / furthermore" elaboration markers
|===

=== L5 ‚Äî Claude-Code tool-native

[cols="1,3",options="header"]
|===
| Rule | Catches
| no-plan-mode-for-risky | Migration / delete / rewrite ask with no "plan first"
| no-task-list-for-multi-step | 3+ action verbs without a TaskCreate / checklist ask
| no-agents-for-parallel-lookup | Multiple independent lookups without parallel agents
| no-role-for-critique | "Review my X" without invoking a role (skeptic / security / reviewer)
| no-panel-for-contested-design | "Which is better / torn between" without brainstorm-panel
| no-workflow-for-fanout | "For each of these 20+ things" without Workflow / parallel agents
|===

=== L6 ‚Äî skill-awareness

[cols="1,3",options="header"]
|===
| Rule | Catches
| no-skill-lookup | "How do I X" / "what's the standard way" without checking existing skills
| pattern-worth-abstracting | "Again / same as before / yet another" ‚Äî rule of three, worth a skill?
| no-skill-composition | Multi-step repeatable ceremony not named as a skill candidate
| no-edit-preference *(v0.20+)* | "Create a new file/script/helper" without an edit-existing preference
|===

== Nudge modes

Four modes control where the nudge appears. Config resolves in order: repo local ‚Üí user global ‚Üí default.

[cols="1,2,2",options="header"]
|===
| Mode | User sees | Claude sees

| `both` (default) | Boxed nudge on stderr (dim area under prompt) | `additionalContext` with guidance
| `silent` | Nothing | `additionalContext` with guidance
| `log-only` | Nothing | Nothing (every fire still logged)
| `inline` | The boxed nudge is rendered as the opening block of Claude's response | Render instruction + guidance
|===

Switch modes conversationally ‚Äî Claude edits the config file:

[source,text]
----
"set prompt-coach to inline"
"set prompt-coach to silent"
"reset prompt-coach mode"
----

== Voice presets and sources (v0.17+)

Two orthogonal knobs shape *how* the coach talks to you.

`voice_preset` chooses the *personality* ‚Äî which set of pre-written phrasings the coach draws from:

* `colleague` (default) ‚Äî friendly, direct, ends on a concrete question. Assumes fluent colloquial English and light idiom. `+"You said 'fix it' ‚Äî what's 'it'?"+`
* `plain` ‚Äî simple English, short sentences, no idioms, no jargon. Written for non-native speakers or anyone who prefers explicit phrasing. `+"Please tell me what 'it' or 'this' is. For example: a file name, a PR number, or an error message."+`

Coverage: L1 + L2 rules (10 rules) ship both presets with 3 variants each. L3‚ÄìL6 rules currently ship `colleague` only; if you pick `plain`, they fall back to `colleague` (no crash, no missing text).

`voice_source` chooses *who authors the sentence at fire time*, independent of preset:

[cols="1,3,2,2",options="header"]
|===
| Source | How | Cost per fire | Determinism

| `static` (default) | Pre-written variants from the catalog. Rotates + follows disclosure levels. | 0 tokens, ~0 ms | Deterministic ‚Äî same rule, same context ‚Üí same text (modulo novelty rotation)
| `llm-compose` | Hook passes rule id + guidance + your prompt shape to Claude in `additionalContext`; Claude writes the nudge fresh, primed to reflect *this specific* prompt with six anti-disagreement guardrails. | +200‚Äì800 ms, ~150‚Äì400 output tokens | Non-deterministic; may feel repetitive-in-a-different-way
| `hybrid` | First fire in the window = static preset (deterministic, teaching-focused). Medium/short refreshers = `llm-compose` (livelier when repeating). | Costs both, sparingly | Mixed
|===

The six anti-disagreement guardrails baked into `llm-compose`: header line verbatim, word-count cap per disclosure level, must not override the rule even if context resolves the referent, must match the preset's voice, must end on a concrete ask when guidance implies one, and prompt shape (first 60 chars) provided so Claude can situate the response.

Switch conversationally:

[source,text]
----
"set prompt-coach voice to plain"
"set prompt-coach source to llm-compose"
"set prompt-coach source to hybrid"
----

The `outcome` log line records the combination per fire (e.g. `nudged:inline:full:v1:p=plain:src=static`) so `/prompt-coach-beta:stats` can mine which combination you converge on.

== Config surface (v0.18+)

By v0.18 the coach's config had grown to 22 keys across 8 categories. `+/prompt-coach-beta:config+` is the structured surface ‚Äî it reads a `CONFIG_SCHEMA` metadata dict alongside `DEFAULT_CONFIG`, so future options are picked up automatically once they land in the schema. The dashboard, describer, and validator never drift.

Nine verbs:

[cols="2,4",options="header"]
|===
| Verb | What it does
| `show` | Categorized dashboard with resolved value + source (default/global/repo) per key
| `show <category>` | One category only (e.g. `show voice`)
| `get <key>` | Resolved value only
| `describe <key>` | Full metadata: type, default, current, choices, example, since-version, description
| `set <key> <value>` | Validate against schema ‚Üí deep-merge write to scoped config
| `reset <key>` | Remove your override, revert to previous layer
| `reset-all` | Wipe the scoped config file (requires confirmation)
| `diff` | Show only keys where the resolved value differs from the default
| `export` | Print the resolved config as JSON (for backup)
|===

Flags: `--scope global` (default) or `--scope repo` chooses which config file writes go to. `--dry-run` on `set`/`reset`/`reset-all` previews without touching disk. `--json` on `show`/`get`/`describe`/`diff` produces machine-readable output.

Categories: `output`, `voice`, `rule-activation`, `mastery`, `praise`, `typo-tolerance`, `anti-habituation`, `llm-fallback`.

**Say-it phrases still work** ‚Äî natural language like `+"set prompt-coach to inline"+` or `+"coach off no-few-shot"+` routes to `:config set` under the hood. The command is the discoverable path when you don't know what the options are called; natural language is the fast path when you do.

Writes are deep-merged into the target JSON so keys the schema doesn't know about (forward-compat entries from future versions) are preserved, not stripped. Invalid values are rejected with a clear error message.

=== Options, interactive flows, and mastery (v0.19+)

Four more verbs joined `:config` in v0.19:

[cols="2,4",options="header"]
|===
| Verb | What it does
| `options <key>` | Enumerates legal values for a key with per-choice explanations. For enums (`nudge_style`, `voice_preset`, `voice_source`), lists each choice + what it does. For int/bool/list keys, shows type + current + default + example + description.
| `quick` | Interactive multi-choice picker for the ~4 categorical settings (`voice_preset`, `voice_source`, `nudge_style`, `praise_ratio`). Claude walks you through with `AskUserQuestion`; each answer routes to `:config set`.
| `full` | Same interactive pattern for every schema key. Enums ‚Üí picker; numeric/bool ‚Üí keep-current / type-new / reset-to-default. Longer flow; you can skip categories.
| `mastery` | Dashboard: mastered rules (with mastery date), in-progress rules (fires_total > 0, sorted by streak descending so closest-to-mastery shows first), dormant count by tier.
| `mastery-reset <rule-id>` | Zeros fires_total, clean_streak, status, mastered_at for one rule. Preserves `prompt_count` + other rules + non-schema keys. Dry-run preview first.
| `mastery-reset-all` | Same across every rule with accumulated state. Preserves `prompt_count`. Dry-run preview first.
|===

Under the hood, enum keys in `CONFIG_SCHEMA` now carry an optional `choice_descriptions` dict ‚Äî same forward-compat pattern as v0.18. Adding a new enum config means adding a `choice_descriptions` dict; `options` and `quick` pick it up automatically.

== Anthropic guide alignment (v0.20+)

v0.20 aligns the coach with Anthropic's live https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices[prompting best-practices guide]. Two-part change:

**Six new rules** closing high-value gaps in the coach's coverage:

* `no-xml-structure` (L3) ‚Äî flags ‚â•7-line pastes without `<code>` / `<data>` / `<logs>` delimiters. Anthropic: `structure-prompts-with-xml-tags`.
* `no-classical-role` (L3) ‚Äî flags critique/review/audit asks (>15 words) that don't invoke a role via "you are a X". Distinct from `no-role-for-critique` which flags the Claude Code `/roles:as` variant. Anthropic: `give-claude-a-role`.
* `test-goalseeking` (L3) ‚Äî flags "make tests pass" / "get CI green" / "fix broken tests" without stated correctness intent. Anthropic explicitly calls this out as a gaming/hallucination vector. `avoid-focusing-on-passing-tests-and-hard-coding`.
* `no-verify-before-claim` (L3) ‚Äî flags assertion-shaped questions ("does X exist", "which files use Y") that don't demand receipts (file:line, quoted code). Distinct from `no-uncertainty-budget`, which is about admitting uncertainty; this is about demanding evidence. `minimizing-hallucinations-in-agentic-coding`.
* `overthinking-warning` (L4) ‚Äî the inverse of the coach's usual push-toward-more-structure direction. Fires on ‚â•60-word prompts with 3+ elaboration markers ("make sure to", "be very careful", "please also", "furthermore", etc.). `overthinking-and-excessive-thoroughness`.
* `no-edit-preference` (L6) ‚Äî flags "create/write/make new script/helper/file" without an edit-existing preference. `reduce-file-creation-in-agentic-coding`.

**`Rule.anthropic_ref` field.** Every rule now optionally carries a section slug from Anthropic's page. 25 of the 34 shipped rules link to a canonical section; 9 are Claude-Code-specific or novel coach concepts with no direct upstream mapping.

**`/prompt-coach-beta:config sources`** ‚Äî new verb that surfaces the citation trail:

[cols="2,4",options="header"]
|===
| Form | What it shows
| `sources` (no arg) | Overview: base URL + every rule grouped by its `anthropic_ref`, plus a separate section for unlinked rules. Header notes coverage (e.g. "25/34 rules link to an Anthropic-guide section").
| `sources <rule-id>` | Per-rule detail: tier, name, `anthropic_ref` slug + full URL to the exact section anchor, and the full list of cited sources (titles + URLs).
| `--json sources` / `--json sources <rule-id>` | Machine-readable equivalents for scripting.
|===

Enables tracing *"why does this rule exist?"* back to Anthropic's authoritative material in one command.

== Encouragement layer

The coach also praises the *specific positive behaviors* that mirror the negative rules ‚Äî 22 positive detectors. Praise is deliberately sparing (default 1 per 10 clean prompts with a positive, plus milestone events on rule mastery and first-after-fire), grounded in behavioral-science literature (Brophy 1981, Mueller & Dweck 1998, Fogg 2019, Deci & Ryan 2000, Kohn 1993).

Design constraints:

* *Never on a nudged prompt.* Praise + correction on the same prompt dilutes both.
* *Specific, not generic.* Describes what you did, not who you are.
* *Process, not trait.* Praises the technique.
* *Variable-ratio.* Intermittent praise stays potent.
* *Milestone events.* Rule graduating to mastered + first-after-fire always recognized.

== Anti-habituation

Same message repeated wears out. The literature (Hattie & Timperley 2007 on feedback wear-out; Sasse & Rashid 2013 on alert fatigue) converges on *variety + frequency capping + escalation* as the combination that keeps coaching salient. The coach ships all four:

* *Variant pool* ‚Äî L1 rules have 3 phrasing variants each. Rotates on each fire.
* *Novelty constraint* ‚Äî never repeats a variant within the last 2 fires of the same rule.
* *Progressive disclosure* ‚Äî first fire in the window shows the full box; 2nd‚Äì3rd is a medium reminder; 4th+ is a one-liner.
* *Silence after saturation* ‚Äî 5 fires within 30 prompts triggers a 30-prompt silence on that rule. Absence signals that coaching isn't landing.

== Typo tolerance

A pre-pass normalizes each prompt against a curated set of trigger words drawn from the rule catalog. Dyslexic-friendly, transposition-friendly, deterministic (banded Levenshtein, ~1 ms). Corrections happen only when the closest trigger word is uniquely the closest (ties = no correction, so `text` doesn't become `test`), and adaptive tolerance is stricter on short tokens.

Examples: `refacotr` ‚Üí `refactor`, `evrything` ‚Üí `everything`, `veryfy` ‚Üí `verify`, `cleanre` ‚Üí `cleaner`.

Set `typo_tolerance: 0` to disable.

== Conversational short-circuit

Many prompts in an ongoing conversation aren't full asks ‚Äî they're approvals (`sure`, `publish`, `go`), multi-choice picks (`1 and 2`, `option a`), or continuations (`continue`, `thanks`). The coach detects these and skips analysis entirely ‚Äî no rule matching, no praise, no streak updates. Task-notification and system-reminder messages are also short-circuited.

== Mastery ‚âÝ silence

Mastered rules aren't permanently dormant. They still evaluate every prompt, and when only a mastered rule matches, a *refresher* fires: a one-line softer box (`üîÑ prompt-coach ‚Äî refresher on <rule>`) rather than the full nudge. The cooldown for mastered rules is much longer (50 prompts vs 5 for practicing rules). If a mastered rule keeps matching, that's the highest-signal event in the system ‚Äî evidence that a habit is regressing.

Optional self-healing: `demote_on_regression` auto-demotes a mastered rule that fires threshold+ times within a rolling window. Off by default.

== Reporting bad calls

If the coach mistreated a prompt, mark it on the very next turn with any of these phrases:

* "coach that was wrong"
* "coach missed this"
* "coach false positive"
* "bad nudge" / "wrong nudge"

The analyzer flags the *prior* substantive prompt's analysis (not the current complaint) into `.claude/prompt-coach/candidates.jsonl`. Later, run `/prompt-coach-beta:report-issue` to review candidates, add annotations, preview a redacted payload, and (only after explicit confirmation) file a GitHub issue.

*Privacy guarantees:* the report includes a structural signature (word count, question-shape, has-file-ref, etc.), the first 5 words of the prompt only, and your annotation. Full prompt content stays local in `log.md`.

== Future ‚Äî Java MCP server

The current Python + `UserPromptSubmit` hook is CLI-only. A Java MCP server is planned so the coach can run in claude.ai chat, aggregate telemetry across users, and support persistent per-user state. The prerequisite is real training data from many users (not a single maintainer's log). The `report-issue` command already produces training-data-shaped payloads.

The living design spec lives in the repo at `plugins/prompt-coach-beta/docs/java-mcp-spec.md` ‚Äî MCP surface, canonical rule catalog format, data model, privacy model, and migration path.

== Design notes

* *Pure heuristics.* Rules are regex + short-window checks. No LLM call per prompt. Cheap, deterministic, ~10 ms.
* *False positives are OK.* Cooldowns prevent pestering; graduation rewards consistent clean prompts even in the face of an over-eager rule.
* *Sources are diverse.* Each rule cites 2+ sources drawn from Anthropic prompt-engineering docs, Claude Code best practices, OpenAI's prompt guide, Simon Willison's notes, and academic work (Wei et al. 2022 on CoT, Brown et al. 2020 on few-shot, Schulhoff et al. 2024 on prompt patterns).
* *Beta.* Graduates to `prompt-coach` (drops the suffix, cuts a `1.0.0` release) once it earns its stable slot.
